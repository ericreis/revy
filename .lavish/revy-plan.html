An agentic PR review tool, built as a local AXI

revy is a CLI that spins up a local browser review surface for a GitHub PR and turns the coding agent that launched it into a live review partner: it recommends where to start, answers questions anchored to selected code in persistent threads, and syncs your review comments back to GitHub's native PR UI. The interaction model is copied from lavish-axi (server + long-poll conversation loop); the diff surface and GitHub sync are the parts lavish does not have.

Why we're building this

You review GitHub PRs and want AI woven into the loop: a better starting order, the ability to select code and ask questions with the conversation kept in-tool, and comments that land in GitHub's native UI. It should run locally, be spun up by a skill, and stay model/agent-agnostic (Claude Code first). We surveyed ~25 tools; none does all of it.

Covered by existing tools

  • Local server + GitHub-style diff (difit)
  • Reads PR via gh
  • Threaded local comments (difit)

The gaps = revy's reason to exist

  • 1 Comments sync to GitHub native UI
  • 2 AI-recommended review order
  • 3 Select code → agent → persistent threads

Why not fork difit?

difit is a monolithic app built for the opposite model - human reviews locally, then copies a prompt to paste into an agent. revy needs a live agent-in-the-loop. Retrofitting that fights difit's grain, so difit stays a reference, not a base.

Architecture at a glance

One detached local server, keyed by PR identity. The human works in a browser diff UI; the launching agent drives the conversation through a long-poll. Nothing embeds an LLM - the agent is the brain (see the open question below).

Lifecycle - concurrency, resume & re-review

Answers to two things you raised: yes, many threads run at once; and a review is never "one-shot" - you relaunch on the same PR and pick up exactly where GitHub left off.

Concurrent threads

You open as many anchored threads as you like without waiting on the agent. Each poll drains the whole queue, so the agent receives a batch and can answer them in parallel via subagents, replying per thread.id. The UI shows each thread's own state - asked agent working answered - and new questions keep queuing while the agent works.

Resume the same PR

Sessions persist and are keyed by PR, so revy owner/repo#1234 again revives the session (exactly how lavish reopens an ended session). On (re)open revy refetches the latest diff + head SHA and re-pulls GitHub review threads, so nothing is lost between sittings.

Re-review after replies

GitHub is the source of truth for synced comment threads; the local file only caches unsent drafts + agent Q&A. On resume, author replies and resolved/unresolved status come straight from GitHub (GraphQL); resolved threads collapse, open ones surface. New commits → new head SHA → refreshed diff, with a heads-up if your drafts now point at changed lines.

Components

CLI (the AXI surface)

Thin command layer any agent can run. Mirrors lavish's verbs.

revy owner/repo#1234   # fetch + open UI
revy poll              # long-poll for human actions
revy reply --thread t_01 "…"
revy order review-order.json
revy submit            # batch → GitHub
revy end / revy stop

Server (express, detached)

  • Loopback bind (127.0.0.1), idle auto-shutdown
  • GET /api/poll long-poll w/ heartbeat (agent)
  • POST /api/:key/actions queue human actions (browser)
  • POST /api/:key/reply agent → thread
  • GET /events/:key SSE live updates (browser)
  • POST /api/:key/submit → GitHub sync

Frontend (React + Vite)

  • react-diff-view: GitHub-style; unified by default (split toggle later)
  • widgets[changeKey] mounts threads + chat under lines
  • App-level shift-click multi-line range selection
  • refractor syntax highlighting

GitHub adapter (via gh)

  • Read: gh pr diff --patch, gh pr view --json
  • Existing threads via GraphQL (isResolved)
  • Write: one PENDING review → N comments → submit
  • Replies + resolve/unresolve; uses user's gh auth

Session model

lavish keeps one chat[] in a single global state.json. revy improves on that in two ways: it uses anchored threads (many concurrent, one per code selection) plus a recommended order and GitHub sync metadata, and it stores one file per PR at ~/.revy/sessions/<key>.json (key = hash of owner/repo#number). Per-PR files keep each file small no matter how many PRs you review, avoid rewriting a large blob on every keystroke, and sidestep write contention when several reviews are open at once.

How the hard parts work

1 · Comment sync - one batched GitHub review

Draft comments live locally; revy submit posts them as a single pending review so the author gets one notification, not N. Multi-line uses start_line/line.

2 · Anchored threads in the diff

react-diff-view's widgets map a change key (e.g. N42) to a React node rendered directly under that line - this is where a comment box or an agent chat thread mounts.

Starting a thread (your question): both kinds begin the same way - select a line or drag a range in the gutter. A + affordance appears (just like GitHub); clicking it opens a small menu with the two actions. That single entry point is what keeps review comments and agent questions living side by side in the same anchored thread list.

The distinction: a review comment is a draft that batches into your GitHub review on revy submit. An agent question opens a live Q&A thread (answered through the poll loop) that stays on revy's side only - it is never posted to GitHub automatically (not on resolve, not on submit). If you decide a takeaway is worth sharing, one click turns it into a review comment; otherwise the conversation stays private to your local review.

Thread style (your earlier question): we mirror GitHub's comment layout, not chat bubbles - every entry is uniformly left-aligned with an avatar, a handle, and a timestamp. That keeps the tool consistent with the native PR UI these comments sync into, and lets review comments and agent Q&A share one visual language. The reviewer shows as their GitHub username; the agent shows as a configurable handle (e.g. claude) with an agent tag.

Why Node/TypeScript and not Go?

The diff surface (react-diff-view) is React, so the frontend is JS/TS no matter what - a Go backend would mean two languages and toolchains for one small local tool. Go's real edges (goroutine concurrency, a single static binary) don't pay off here: there's one local user with a handful of long-poll connections, and a /revy + npx skill would give zero-setup distribution later if we ever want it. Staying in one language lets us lift lavish's proven server/poll code almost verbatim - the bigger maintainability win. The GitHub layer is a gh subprocess, so it's language-agnostic; if we ever needed a standalone binary we could reimplement just the thin server in Go behind the same HTTP contract.

Risks & how we handle them

Diff line → GitHub params. react-diff-view gives old/new line numbers; GitHub needs line/side/start_line. We own a small mapping + validate the target is in the diff before posting.
Stale commit_id. If the PR is pushed to mid-review, comments can reject. Refetch head SHA on submit; warn if the diff changed.
Secondary rate limit (~80/min). Batching as one pending review is the mitigation; respect Retry-After.
Agent presence. Like lavish, the UI shows when no agent is polling; queued questions/comments persist until the next poll.
